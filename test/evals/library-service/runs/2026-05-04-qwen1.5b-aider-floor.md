## What actually happened

Both conditions failed to produce compilable Go code matching the seed's
project structure.

**Baseline (1m23s)**:
- S01: 66s, 2 files, build=fail tests=fail
- S02-S05: 3-4s each, 0 files. Aider exited fast — likely qwen produced
  no parseable Edit blocks.

**Cortex (1h33m)**:
- S01: 57s, 2 files, build=fail tests=fail
- S02: 43s (preamble 700 B), 3 files, build=fail tests=fail
- S03: 20s (preamble 2260 B), 0 files, build=fail tests=fail
- S04: **5,467,372 ms = 91 minutes** (preamble 2260 B), 0 files, build=fail.
  Aider waited on qwen entering a degenerate generation loop that never
  produced a parseable response.
- S05: 14s (preamble 2311 B), 0 files, build=fail tests=fail

Sample output qwen produced (cortex S01 / S02):
- Imports `github.com/lib/pq` (PostgreSQL driver) but tries to open
  SQLite — mixed-up driver knowledge.
- Uses `sql.Open`, `json.NewEncoder`, `log.Printf` without importing
  `database/sql`, `encoding/json`, or `log`. Build fails on missing
  imports.
- Created a file at literal path `path/to/cmd/server/main.go` —
  interpreted "path/to/" from a templated example as a real
  directory name. Classic small-model placeholder confusion.
- Left `cmd/server/main.go` as the empty seed; created handlers at
  the workdir root instead of following the seed's `cmd/server/`
  layout.

## Interpretation

**This is not a Cortex-amplifier failure. It's a model-floor failure.**

The amplifier thesis presupposes the model can do the task in the first
place; Cortex makes it do the task *more cohesively*. Qwen 1.5B can't
even produce compiling Go code for a multi-file HTTP service from a clear
spec — there's no signal for Cortex to amplify. Even a perfect 700-byte
preamble would not have stopped qwen from hallucinating `lib/pq` while
trying to use SQLite.

Cortex's preamble was *somewhat* visible in the cortex condition's
slightly-better S02 output (3 files vs baseline's 0), but not enough to
overcome the model's underlying capability gap. By S03 onward both
conditions were producing nothing.

## Sharpens the thesis

> Cortex amplifies models that are at or above the capability threshold
> for the task.

The threshold for *this* task (multi-file, multi-package Go HTTP service
with handlers, storage, persistence, tests) is somewhere **between qwen
2.5-coder:1.5b and Haiku 4.5**. Probably much closer to Haiku — Haiku
crushed it; qwen 1.5B couldn't approach it.

Useful followups to find the actual floor:

- **Bigger local model**: `qwen2.5-coder:7b`, `deepseek-coder-v2:16b`,
  `codellama:13b`. Each step up roughly 5× larger and 5-10× slower
  per session, but should clear the floor for this task class.
- **Smaller task scope**: rewrite the eval for one resource only.
  If qwen 1.5B can do *one* CRUD resource cohesively, the floor for
  cohesion-amplification work is lower than the multi-file project
  floor.
- **Different harness**: aider's `whole edit format` requires the model
  to output entire files in one shot. A structured-edit format (Plan
  mode, smaller diffs) might let smaller models contribute usefully
  even where they can't produce a whole file.

## What this run is good for

- **Floor data point**: confirms qwen 1.5B is below the floor for this
  task. Don't re-run on this model class for this task.
- **Failure-mode catalog**: the specific things qwen got wrong
  (literal `path/to/`, missing imports, mixed DB drivers) are the
  symptom set future thesis tests should check for as "model is
  below floor" vs "Cortex isn't helping" — those are different
  problems with different fixes.
- **Wall-clock budget reality check**: even when failing, qwen took
  1.5 hours for the cortex condition because of one degenerate session.
  Real qwen 7B / 13B runs will take longer, plan accordingly.
