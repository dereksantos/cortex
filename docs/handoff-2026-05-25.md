# Handoff — Per-node routing landed; chunker pathology is next

> **Date:** 2026-05-25
> **Branches landed:** PR #54 (per-node routing), PR #55 (routing tuning + XML tool-call recovery + intent-aware no_progress)
> **Current branch:** `derek.s/chunker-and-handoff` (this doc lives here)
> **Active fleet:** `chatterbox:4000` (LiteLLM proxy) serving `coder` (Qwen3-Coder-30B), `reasoner` (gpt-oss-20b), `xlam-1b-fc-r`, `embedder`, `reranker`. Operator config in `.cortex/config.json` with endpoint `Models` list — no `routing` overrides needed (auto-detection carries it).

## What's working end-to-end

Verified live against the chatterbox fleet:

- **Routing layer**: every node lands on the right model. `decide.tool_call` → xLAM, `decide.next` → reasoner (gpt-5.4 from OpenRouter on the user's stack, auto-tagged `CapReasoning`), `sense.classify_intent` → coder, `decide.coding_turn` → coder.
- **Inner agent loop**: the coder emits XML-style tool calls (`<function=name><parameter=k>v</parameter></function>`); the permissive parser in `pkg/llm/xml_tool_calls.go` recovers them; the loop dispatches as if they were OpenAI structured `tool_calls`. First end-to-end successful task in the project.
- **Intent-aware termination**: `Loop.Intent` gates the `no_progress` heuristic's condition 1. Review / recall / meta sessions can do read-only exploration without being killed at turn 5. Code intent retains the old behavior.
- **Trailing-K detection**: `no_progress` condition 2 catches "model gets stuck re-reading the same file" even when earlier turns had diverse reads. K=4 — 3 consecutive identical reads can be legitimate (mining a long file for symbols); 4 is hard to explain otherwise.
- **JSONL telemetry**: `dag_traces.jsonl` now carries `picked_model` and `picked_reason` per row. Diagnosable.

## The remaining blocker — root cause identified

The harness is now usable for SHORT exploration, but on bigger projects the coder gets stuck in a **re-read loop**. The empirical evidence:

```
turn 4 → read_file(cmd/cortex/commands/repl.go)   16336 chars
turn 5 → read_file(cmd/cortex/commands/repl.go)   16336 chars (same)
turn 6 → read_file(cmd/cortex/commands/repl.go)   16336 chars (same)
turn 7 → read_file(cmd/cortex/commands/repl.go)   16336 chars (same)
```

Root cause is **not** model confusion. From `.cortex/db/dag_traces.jsonl`:

```
chunk-8  emitted=8 of 35  max_tokens=500
chunk-10 emitted=8 of 35
chunk-12 emitted=8 of 35
chunk-14 emitted=8 of 35
```

And from `pkg/cognition/dag/salience.go:209-214`:

```go
const MaxEmittedChunks = 8
```

`repl.go` is 4300 lines → splits into 35 chunks at the 500-token salience cap. The chunker truncates emission at 8 chunks (~22% of the file). The truncation marker says **"re-fetch with explicit line range"** — but `act.read_file`'s tool surface only accepts `path`. **The chunker's advice is unreachable.** The model is following the marker's instruction by re-fetching, and getting the same 22% slice every time. It's not the model's fault; it's a contradiction in the surface design.

## Next session — two pieces of work, ordered

### Piece 1 (quick, immediate value): raise the chunker cap, intent-aware

The 8-chunk cap was sized for the old salience cap range (1K-8K tokens). At the current 500-token cap it's pathologically small. The cleanest fix is **cap by total emitted tokens, not chunk count**, and make it intent-aware (same plumbing pattern we used for `Loop.Intent`):

- Add `SalienceContract.MaxEmittedTokens int` field (0 = default 4000, same effective limit as today).
- In `applyDeterministicChunking` / `ChunkOversize`, replace the chunk-count cap with a token-budget cap.
- In `decide.tool_call`'s spawn, set `MaxEmittedTokens` from the session intent:
  - `code` → 4000 (current; focus)
  - `review` / `recall` / `meta` → 16000 (covers ~90% of real files entirely)
- Hard ceiling: don't exceed ~30% of the model's context window (`Budget.MaxContextTokens × 0.30`). For coder at 65K that's ~20K tokens; 16K stays under.

Simpler alternative if intent threading feels too heavy: just bump `MaxEmittedChunks` from 8 to **32** unconditionally. One-line change, makes the harness usable today, costs ~10-20s of additional accumulator latency on big files when code-intent sessions touch a large file.

Tests:
- Pin that current behavior (4K cap) still applies on `code` intent.
- Pin that `review` intent emits more chunks for a multi-chunk file.
- Pin that the hard context-window ceiling caps things on small-context models.

### Piece 2 (architectural, real fix): add `start_line` / `end_line` to `act.read_file`

This is the honest tool surface. The chunker already TELLS the model what line range it's missing — the surface should let the model ask for it.

- Add `start_line` and `end_line` (1-indexed, optional) to `act.read_file`'s `Inputs` schema.
- Update the tool handler (`internal/harness/readfile.go` or wherever `NewReadFileTool` lives) to slice the file by line range when present.
- Update the truncation marker emitted by `ChunkOversize` to suggest a concrete next-range call (e.g. `re-fetch with start_line=4001, end_line=4500`).
- The `tools.json` xLAM sees needs the new params; xLAM has to actually do the line math (it might or might not — this is the empirical question, run it to find out).
- Decision needed: should the chunker still cap on first read (forcing the model to paginate) OR should it emit everything when the model can paginate? **My take: still cap.** The cap is the accumulator's protection. Pagination is for when the model wants more, not as the default first-read.

Tests:
- Read with no line range: chunked emission as today.
- Read with `start_line=100, end_line=200`: returns exactly those lines.
- Read with `start_line=4000, end_line=5000` on a 4300-line file: returns lines 4000-4300 (clipped, no error).
- Tool surface advertised in `tools.json` includes the new params.

### Piece 3 (background, lower priority): why the model didn't escape with `list_dir`

In the failing session, after re-reading `repl.go` four times, the model didn't fall back to `list_dir` for adjacent files. The accumulator snapshot wasn't telling it "you already have repl.go; explore elsewhere." Worth investigating after Piece 1+2:

- Dump the session's accumulator snapshots (`.cortex/sessions/<latest>/session.jsonl`) and read what the model actually saw on turn 5-7.
- Does the snapshot make clear "you have read these files"? Probably not — it compresses *content* not *what was read*.
- A small change to the accumulator prompt could add a "files already read" header. Candidate: `pkg/cognition/dag/ops/attend_accumulate.go`.

## Things to keep in mind

- **Don't add `chatterbox` to the code.** It's the user's machine name, not a hardcoded fleet identifier. All endpoint references should go through `cfg.Endpoints` / `cfg.ResolveModelRoute`.
- **`MaxOutputTokens` on the salience contract is the per-chunk size cap**, not the total emitted budget. They're different. Don't conflate.
- **The reranker (`/v1/rerank`) is still not wired.** `attend.rerank` does LLM-based reranking via chat completions; a dedicated reranker client + interface is a separate piece. Not blocking.
- **Re-running the test prompt** (`Can you explain some of the core files in this project?`) is the canonical regression check. Before today's work it terminated with `no_progress` at turn 5; after Piece 1 it should complete with a real synthesis of multiple files.
- **The full trace lives in `.cortex/db/dag_traces.jsonl`** and now carries `picked_model` + `picked_reason`. The session's per-turn JSONL is in `.cortex/sessions/<TIMESTAMP>/session.jsonl`. Both worth tailing while iterating.
