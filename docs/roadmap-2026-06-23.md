# Roadmap Analysis & Plan — 2026-06-23

> Four roadmap items, grounded in the current codebase. This document
> is the **first tidy pass**: it names what exists, what each item
> actually requires, and the smallest principled slice for each — in
> dependency order. It is written to be committed as a planning
> artifact, not a spec.

---

## 0. The lay of the land

The `cmd/loop` harness is the live product surface. It is a **4,371-line
`main.go` monolith** (`cmd/loop/main.go`) that owns, in one file:

- The system prompt (`SystemPrompt`, L69) + `AGENTS.md` injection
  (`projectInstructions`, L2370)
- Tool declarations and the `Tool`/`ToolFunction` schema helpers
  (`readFile`, `writeFile`, `editFile`, `studyTool`, `projectIndexTool`,
  `bash`, `removeTool` — L521–649)
- The `ToolCall.Execute` dispatcher + per-tool implementations
  (`ReadFile`, `WriteFile`, `EditFile`, `Study`, `ProjectIndex`,
  `Bash`, `RemovePath` — L938ff)
- The `CortexSession` state machine: transcript, retrieval, capture,
  distillation, compaction, metrics (L1939ff)
- The agentic inner loop `Resolve` + `runToolCalls` (L3809ff)
- The `Turn` entry point shared by REPL + headless + Discord (L3908)
- The REPL `main` loop + slash commands (L4077ff)
- Streaming/render (`streamPrinter`, `render.go`)
- Fleet/config/model selection (L2069ff, L2223ff)

There is a **parallel, cleaner harness** in `internal/harness/`
(`loop.go`, `tools.go`, `tool_*.go`) used by the eval grid. That one
has a proper `ToolHandler` interface, a `ToolRegistry`, a `Loop` with
budget/no-progress guards, and an `AccumulatorSnapshot` bounded-context
rewrite. The `cmd/loop` harness predates it and does **not** share its
abstractions — `cmd/loop` has its own `Tool`/`ToolCall`/`Message` types
that duplicate `llm.ToolSpec`/`llm.ChatMessage`.

This duplication is the central design debt. It is also the lever:
every roadmap item is easier to land cleanly in `cmd/loop` if the tool
surface is a small, well-factored layer rather than 1,600 lines of
inline methods on `ToolCall`.

### What's already designed (docs)

- **Working memory / forever session** — `docs/prompts/forever-session-design.md`
  is the design. Its central claim: **the study engine is not a tool
  that compaction calls — it *is* the compaction mechanism.** The
  current `cmd/loop` `Compact` already calls `compactStudy` → `runStudy`
  (the study engine). The forever-session design's innovation is making
  that compaction **incremental** (study the last K turns, not the
  whole transcript) and **layered** (a persistent session-state message
  that survives across compactions via study-based merge). `docs/working-memory.md`
  is the companion: the keep/compress/evict triage framing, the
  "window is a cache, journal is memory" reframe.
- **Tool surface** — `docs/tool-surface.md` names six axes (contract,
  authorization, dispatch, result, state, observability) and the
  current gaps. The `cmd/loop` tools satisfy none of them uniformly.
- **Shell risk** — `internal/shellrisk` classifies commands; network
  calls (`curl`/`wget`/`fetch`) are in the gray-zone-to-deny floor.
  This is directly relevant to a web search tool: the harness already
  has a risk gate for outbound network, and it currently **denies** it.

## 1. Web Search tool

### What it is

A `web_search` tool the model can call during a turn, plus a
`fetch_url` companion for retrieving a specific URL's content. The
model gets current information (docs, APIs, error messages, library
versions) without the user pasting it.

### What exists

- `remember.fetch_external` (`pkg/cognition/dag/ops/`) — but it only
  shells out to `go doc` for local Go packages. The doc explicitly
  parks network fetching: "a future iteration can add an HTTP client;
  the journal's local-only invariant gets a sibling 'external lookups
  are opt-in and logged' rule."
- `internal/shellrisk` — network calls are classified as
  gray-zone-to-deny. `curl|sh` and `curl|python` are in the deny
  floor. A bare `curl` lands in the gray zone (needs approval).
- No HTTP-fetching tool exists in either harness.

### The smallest principled slice

**One tool, one provider, opt-in, logged.** Not a general browser.

1. **`web_search` tool** in `cmd/loop` — args `{query, max_results}`.
   Backed by a single search provider behind an interface so we can
   swap (start with an HTTP API: DuckDuckGo Lite HTML scrape, or
   SearXNG if the user runs one — both free, no key). Return
   `{title, url, snippet}[]`, capped at `maxResults` (default 5),
   each snippet truncated to ~300 chars. Total output bounded by
   `maxToolOutput` (10k chars, the existing cap).
2. **`fetch_url` tool** — args `{url}`. HTTP GET, content-type sniffed,
   HTML → text extraction (a small `stripTags` — no full browser),
   markdown/JSON/text passthrough. Capped at `curationBudgetTokens`
   (16k, the existing read threshold) so a fetch can't blow context;
   over-cap redirects to `study` on the fetched file (write to a temp,
   study, discard) — reusing the existing curation path.
3. **Risk gate integration** — both tools are **opt-in**:
   `CORTEX_LOOP_WEB=1` env flag, or a config field. Default off. When
   on, every call writes a journal entry (a new
   `external_lookup` kind) so the local-only invariant gets its
   "opt-in and logged" sibling per the `fetch_external` doc's stated
   future. The shell-risk gate does **not** see these — they're a
   dedicated tool with their own authorization axis (axis 2 of
   `tool-surface.md`), not a shell command.
4. **No browser, no JS, no auth.** The first pass is read-only HTTP
   fetch + text extraction. If the user needs JS-rendered pages,
   that's a later bet (a headless browser is a large surface and a
   security liability).

### Where it lands

The tool definitions live in `cmd/loop/main.go` today. The **first
tidy step** (item 2 below) is to extract the tool layer into a
`cmd/loop/tools/` package so `web_search` isn't another 200 lines
inlined into the monolith. If that extraction hasn't happened, the
web search tool is a 200-line inline `ToolCall.WebSearch` method —
workable but deepens the debt.

**Recommendation:** land the tool-layer extraction (item 2, slice A)
**first**, then add `web_search` as a clean new file in the extracted
package.

## 2. Tidy first + small-batch system prompt

### What it is

Two intertwined asks:

**(a) Tidy first — the codebase.** Refactor `cmd/loop/main.go` so the
codebase is clean before adding features. Follow good development
principles and commit hygiene.

**(b) Small batches + tidy first — the agent's behavior.** Tweak the
cortex **system prompt** so the agent itself favours working in small
batches to deliver valuable checkpoints, and keeps the codebase clean.
This is prompt engineering, not harness mechanics.

### (a) Tidy first — the extraction plan

`cmd/loop/main.go` is 4,371 lines. The principled cut is by
responsibility, not by line count:

| New package | What moves out | Lines (approx) |
|---|---|---|
| `cmd/loop/tools` | `Tool`/`ToolFunction`/`newTool`/`objectSchema`/`stringProp`/`boolProp`, the tool `var` declarations, `ToolCall.Execute` + all per-tool methods (`ReadFile`, `WriteFile`, `EditFile`, `Study`, `ProjectIndex`, `Bash`, `RemovePath`), `printToolAction`, `parseXMLToolCalls`, `stripToolMarkup`, `gateShell`, `confinedPath`, the edit helpers (`applyEdit`, `tolerantEdit`, `windowMatches`, `rebuildWithReplacements`, `swapIndent`, `nearMissHint`, etc.) | ~1,600 |
| `cmd/loop/session` | `CortexSession`, `NewCortexSession`, `Append`, transcript (`Start`/`Resume`/`writeEntry`/`writeTranscript`), retrieval (`EnableRetrieval`/`retrieve`/`formatRetrieved`), capture (`captureTurn`/`noteTurn`), distillation (`startDistill`/`distillLoop`/`distillPending`), `Compact`, `contextRatio`, `sessionSummary`, `emitSessionMetrics`, `Close` | ~1,400 |
| `cmd/loop/transport` | `AgentRequest`/`AgentResponse`/`Message`/`ToolCall`/`Usage`/`Choice`, `Send`/`sendOnce`/`SendStream`/`assembleStreamResponse`, `wireMessages`, `applyPromptCache`, `parseCtxSize`, retry logic | ~700 |
| `cmd/loop/fleet` | `Fleet`/`ModelSpec`/`ModelInfo`/`Backend`/`Config`/`ToolConfig`, `discoverFleet`/`applyFleet`/`selectModel`/`resolveBinding`/`LoadConfig`/`findConfigPath`/`projectInstructions` | ~500 |
| `cmd/loop/render` | already exists (`render.go`) — absorb `streamPrinter`, `Message.render`/`Print`/`gutter`, `Prompt`, `ctxColor`, `humanK`/`humanCost` | ~300 |
| `cmd/loop/main.go` (what's left) | `main`, the REPL loop, slash commands, `Turn`, `Resolve`, `runToolCalls`, `runAnchoredTurn`, `compactNow`, the CLI entrypoints (`runStudyCLI`/`runTurnCLI`/`runStudyEval*`) | ~800 |

**Commit hygiene:** each extraction is **one commit**, in this order:
1. `tools` (no behavior change — pure move; the `ToolCall` methods
   become free functions taking a `*CortexSession` or a small
   `ToolCtx`).
2. `transport` (the wire types + send).
3. `fleet`/`config`.
4. `session` (the big one — transcript/retrieval/capture/distill).
5. `render` absorbs the display code.

Each commit: `go build ./... && go test ./cmd/loop/...` green. No
behavior change. The `main_test.go` (2,394 lines) is the safety net —
it tests the public surface, so a pure move that breaks wiring fails
fast.

**The key design decision:** `cmd/loop` keeps its **own** tool types
(`Tool`, `ToolCall`, `Message`) rather than adopting
`internal/harness`'s `ToolHandler`/`llm.ToolSpec`. Why: the `cmd/loop`
harness has features the eval harness doesn't (streaming, live anchor,
distillation, retrieval injection, the `study`-as-compaction path).
Merging them is a larger bet than this roadmap. The tidy pass
**reduces the monolith** without a risky cross-harness merge. A future
roadmap item can evaluate unifying on `internal/harness` once the
`cmd/loop` side is clean and the feature gap is explicit.

### (b) Small batches + tidy first — the system prompt

The current `SystemPrompt` (L69) is one sentence:

> Your are cortex, a coding agent focused on a continous quality
> improvement approach that achieves goals by working towards the
> simplest principled implementation that follows good system design
> and code design. Use your best judgement to make sound decisions
> that favour excellent outcomes over time. Use the provided tools to
> inspect files before answering.

It says nothing about *how* to work — batch size, commit hygiene,
leaving the codebase clean. The agent inherits whatever habits the
model has. The ask is to steer it explicitly.

The change is to extend `SystemPrompt` (and the `compactGoal` that
steers what survives compaction) with a **working-style** section. The
principles to encode, drawn from the project's own `CONTRIBUTING.md`
discipline:

- **Small batches.** Prefer a sequence of small, reviewable changes
  over one large change. Each change should leave the build green and
  the codebase in a working state. Stop and report at a natural
  checkpoint rather than batching unrelated work into one turn.
- **Tidy first.** Before adding a feature, make the change easy (rename,
  extract, remove dead code). The tidy commit is separate from the
  feature commit. If a change is hard, the code isn't ready for it.
- **Commit hygiene.** One logical change per commit. A commit compiles
  and passes tests. The message names what and why, not how.
- **Inspect before answering.** (Already present — keep it.)

This is **non-coercive**: the agent still has autonomy. The prompt
steers toward a discipline; it doesn't force a specific batch size or
forbid large changes when they're warranted. The `compactGoal`
(L136) is updated in parallel so a compaction preserves the
"current checkpoint" and "what's tidy vs. what's the feature" framing
— the session state that survives a compaction should carry the
batch/commit intent forward, not just a content summary.

**Where it lands:** `SystemPrompt` and `compactGoal` constants in
`cmd/loop/main.go` (or the extracted `session`/`fleet` package after
2a). A small, high-leverage change. The `AGENTS.md` injection path
already exists, so per-project overrides of the working style are
already supported — the system prompt is the default, `AGENTS.md` is
the project-specific layer.

## 3. Working memory / forever session

### What it is

Continuous context curation for the entire session — the window is a
managed working set, not an append-only log. The session runs
indefinitely on a fixed context window.

### What exists — and why study is the mechanism

The design is settled in `docs/prompts/forever-session-design.md`. Its
central claim:

> **The study engine is not a tool that compaction calls — it *is* the
> compaction mechanism.** The same primitives that study files (bounded
> sampling, provenance-constrained inference, curator-driven deepening)
> study transcripts.

The current `cmd/loop` `Compact` (L2776) **already** calls
`compactStudy` → `runStudy` (the study engine). So the study-based
compaction mechanism is already wired — what's missing is the
**incremental** and **layered** structure the forever-session doc
specifies.

### Why not AccumulatorSnapshot

`internal/harness/loop.go` has an `AccumulatorSnapshot` bounded-context
rewrite: it stubs older tool results and injects a synthesized
working-memory message each turn. It's a good mechanism, but it solves
a **different, narrower** problem than the forever-session design:

- `AccumulatorSnapshot` is **per-turn, mechanical** — it bounds the
  window *between* compactions by stubbing old tool output. It does not
  curate, summarize, or persist a session state. It's a cache-pressure
  valve, not memory.
- The study-based `Compact` is **episodic, LLM-curated** — it replaces
  history with a grounded, cited digest. It *is* memory.

For `cmd/loop`, the study approach is the right one because (a) it's
already the compaction mechanism there, (b) the design doc is written
for it, (c) it carries the curation quality (citations, curator-driven
deepening) that a mechanical stub can't. `AccumulatorSnapshot` is the
right tool for the `internal/harness` eval loop, where there's no
study engine wired and the goal is bounded context for a fixed-length
eval cell — not a forever session.

**Decision: use the study-based incremental compaction from
`forever-session-design.md`. Do not port `AccumulatorSnapshot`.**

### The smallest principled slice

The forever-session doc's Phase 1–2, scoped to what makes the session
feel infinite without the full triage machinery:

1. **`CompactRecent` — incremental compaction.** Today `Compact`
   studies the *entire* transcript. The change: study only the **last
   K turns** (e.g. 50), produce a focused summary, and replace *those*
   turns in the live window — not the whole history. Older turns are
   already gone (prior compactions) or stay. This keeps the compaction
   call **constant-cost regardless of session age** (the doc's key
   cost win: a 10,000-turn session compacts as fast as a 100-turn one).
   The study engine's auto-density + curator-driven DONE already handle
   "how much to summarize" — no fixed k to tune.
2. **Session-state layer.** A persistent user message
   (`[Session state — summary of prior work]`) that survives across
   compactions. Each `CompactRecent` merges its digest into the
   session state via a study call on the combined text (current state
   + new summary), rather than replacing it. The study engine's
   citation validation prevents the merge from introducing
   contradictions — uncited claims drop. This is the doc's "Layer 3":
   long-term memory that grows slowly via merge, never explodes.
3. **Session manifest (fast-load).** A small JSON file
   (`.cortex/session_manifest.json`: task, decisions, active files,
   unresolved, last turn) written after each merge and loaded at
   resume — so a restarted session immediately knows what it was
   doing, without re-studying the transcript.

This is the doc's Phase 1 (session manifest) + Phase 2 (StudyLoop
compaction) + Phase 3 (study-based merge), scoped to the
**incremental + layered** structure. What it deliberately defers:

- **Keep/compress/evict triage** (`working-memory.md` Phase 2) — the
  selective per-item curation. The first slice compacts the whole
  recent window uniformly via study; the triage that keeps high-value
  items verbatim is a quality layer on top, landed after incremental
  compaction is proven.
- **Background curation in Think/Dream** (`working-memory.md` Phase 3)
  — running the triage on spare cycles. The first slice compacts
  episodically (at the boundary / size trigger), same as today.
- **Retrieval-recall of evicted items** — the journal has every turn;
  wiring `vector_search` to recall compacted-out detail is a Phase 3
  follow-up.

The slice is the **incremental + layered** part — the precondition
that makes the session feel infinite. The **curated** part (triage,
background, recall) is the quality layer on top, landed once
incremental is proven in `cmd/loop`.

## 4. Show diffs in edits and other UX improvements

### What it is

When `edit_file` runs, show the user a diff of what changed. Today
`printToolAction` prints `edit_file(path, 2 edits)` — a one-line
action label. The actual change is invisible until the user reads the
file.

### What exists

- `EditFile` (`cmd/loop/main.go` L1277) — applies edits in memory,
  writes the file, returns `edited path (N replacements)`.
- `printToolAction` (L1187) — prints `  ▸ edit_file(path, args)`.
- No diff rendering anywhere in the codebase. `git diff` is available
  via `bash` but the harness doesn't invoke it on edits.

### The smallest principled slice

**Show a unified diff after each `edit_file` call.** Not a full diff
library — a small, dependency-free diff.

1. After `EditFile` applies edits and before it returns, compute a
   line diff between the old content (read before edits) and the new
   content. A simple O(N*M) Myers or even a line-by-line
   longest-common-subsequence diff is fine — edit_file edits are
   small (the model sends a few lines of context). Cap the diff
   output at ~40 lines so a large replace_all doesn't flood the
   terminal.
2. Print it with the existing color helpers: `+` lines in green, `-`
   lines in red, context in gray. One hunk per edit. For the `edits`
   array, one diff per edit, labeled `edit 1/3`, etc.
3. This is **display only** — the tool result to the model is
   unchanged (`edited path (N replacements)`). The diff is for the
   human watching the terminal. In headless/`--json` mode, the diff
   rides in the tool-result metadata (a `diff` field in the JSON
   envelope) so a driver can render it.

### Other UX improvements (same slice, small)

- **`write_file` byte count → line count.** Today: `write_file(path,
  1234 bytes)`. A byte count is unreadable. Show `write_file(path, 42
  lines)` — and for an overwrite of an existing file, show the delta:
  `write_file(path, 42 lines, was 38)`.
- **`bash` exit code on non-zero.** Today non-zero exits are
  observations. Print the exit code in the action label:
  `bash(go test ./...) → exit 1` in yellow, so a failure reads as a
  failure, not a mystery.
- **Tool action timing.** `study` can take seconds. Print the
  duration after the action: `study(cmd/loop) · 4.2s`. A one-line
  addition to `stopActivity`.

These are all **display-layer** changes in `printToolAction` and
`runToolCalls`. They don't touch the tool logic or the wire format.
They're the lowest-risk slice in the roadmap.

## Dependency order

```
2a (extract tools) ──────► 1 (web_search)
                         │
2a (extract tools) ──────► 4 (diff in edits)  [can land before or after extraction]
2a (extract session) ────► 3 (working memory) [incremental Compact needs the session layer clean]
2b (system prompt) ────── independent, can land anytime
```

The **first commit** is the tools extraction (2a, slice A). It
unblocks web_search (1) and makes the diff work (4) land in a clean
file. The session extraction (2a, slice D) unblocks the
working-memory port (3). The system-prompt tweak (2b) is independent
and can land first — it's the cheapest signal that the agent is
behaving in small batches.

### Recommended sequence

1. **2b: system prompt — small batches + tidy first.** Cheapest,
   highest-signal. Lands first so the agent behaves well while the
   codebase is being refactored under it.
2. **2a-A: extract `cmd/loop/tools`** — pure move, no behavior change.
3. **4: diffs in edits + UX** — lands in the freshly-extracted tools
   package. Small, visible, morale.
4. **2a-B: extract `cmd/loop/transport`** — wire types out.
5. **1: web_search** — new tool in the clean tools package.
6. **2a-C: extract `cmd/loop/fleet`** — config/models.
7. **2a-D: extract `cmd/loop/session`** — the big move.
8. **3: working memory — incremental study compaction** — `CompactRecent`
   + session-state layer + manifest, in the clean session layer.

Each is a checkpoint. Each is reviewable. Each leaves the build green.

## What this plan deliberately defers

- **Unifying `cmd/loop` and `internal/harness`** — larger than this
  roadmap; the feature gap (streaming, distillation, retrieval) is
  real. Tidy first, merge later.
- **Working-memory triage / background curation / retrieval recall**
  (`working-memory.md` Phase 2–3) — the incremental + layered
  compaction is the precondition; the curated layer is a separate bet
  once incremental is proven in `cmd/loop`.
- **A real browser / JS rendering for web search** — read-only HTTP +
  text extraction is the first pass; a headless browser is a large
  surface and a security liability.
